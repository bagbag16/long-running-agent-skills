# ACH 设计约束记录

本文件记录作用于特定层级、分支、文档或产物表面的设计约束。
它不是用户长期理念清单，也不是通用偏好库。

设计约束必须绑定明确对象，例如：
- 某一层规则
- 某条架构分支
- 某份文档
- GitHub 首页
- skill 入口说明
- 状态模板

## 什么值得记录

一条设计约束应满足：
- 有明确适用对象
- 会持续影响后续写法、结构、判断或验收
- 不记录会导致同类问题反复出现
- 能转化为可检查的标准

不应记录：
- 一次性措辞意见
- 没有对象的抽象偏好
- 只表达情绪强弱、但不能指导后续判断的意见
- agent 自己推测出的用户长期理念

## 记录格式

每条约束至少包含：

```text
对象：
约束：
原因：
检查方式：
状态：
来源：
```

可选补充：
- 不适用范围
- 与旧约束的关系
- 冲突处理方式

## 当前约束

### DC-001: 外化规则必须可执行

对象：`references/cca/outer.md` 与外化写入判断  
约束：外化规则不能只写“高影响”“必要”这类抽象词，必须给出判断门槛、失败例子和归属对象。  
原因：过泛规则容易让 agent 执行不到位，继续把外化变成主观整理。  
检查方式：读者应能回答“为什么必要”“不写会失败什么”“写到哪里”“当前效力是什么”。  
状态：`active`  
来源：用户指出“需要写得更有效，不能泛泛而谈，需要多补充例子”。

### DC-002: 设计约束必须绑定层级或产物

对象：本文件与后续设计约束记录  
约束：设计约束记录的是某层级、分支、文档或产物的要求，不记录抽象长期理念。  
原因：长期理念过宽，会污染所有后续任务；绑定对象后才可检查、可替换、可关闭。  
检查方式：每条设计约束都必须能说出适用对象和不适用范围。  
状态：`active`  
来源：用户修正“设计约束记录不是对话中投射的长期理念，而是针对一些层级的内容”。

### DC-003: 规则表达要清晰、有力、不复杂

对象：ACH 内部规则文档，尤其是 `inner.md`、`outer.md`、`architecture-branch-map.md`  
约束：表达应清楚、判断有力、复杂度受必要性约束；有必要的实体应加入，没必要的实体应删除。对 `inner` 的行为规则改动，优先收束或替换重叠规则，不追加并列原则。  
原因：规则如果软、泛、绕，会在执行时失效；但过度拆分会增加维护负担。  
检查方式：每个新增概念都应说明必要性；每条规则都应能转化为执行动作或自检问题；若存在重叠规则，应能说明为什么不合并。  
状态：`active`  
来源：用户对奥卡姆剃刀的修正，以及对规则表达力度和繁杂度的要求。

### DC-004: GitHub 入口保持单一产品身份

对象：README、SKILL.md、docs 与示例  
约束：项目对外讲述应围绕 Agent Continuity Harness / ACH；ADG 与 CCA 只作为内部模式解释，不拆成多个用户侧产品。  
原因：多入口会削弱项目理解、安装决策和首页叙事一致性。  
检查方式：用户从 GitHub 首页应先看到 ACH 是什么、解决什么、如何安装使用，而不是被要求选择内部模块。  
状态：`active`  
来源：用户质疑“为什么做成四个 skill，而不是在 ACH 基础上加内容”。

### DC-005: 同窗口治理先于交接治理

对象：same-window continuation 与 handoff continuation 分支  
约束：先解决当前窗口内必要内容如何升格、如何避免重复全盘阅读、如何处理新旧冲突，再扩展到 handoff。  
原因：如果当前窗口内状态都不清楚，交接摘要只会压缩混乱。  
检查方式：handoff 规则不得绕过 `outer`，也不得把未外化内容只靠摘要承接。  
状态：`active`  
来源：用户判断“先做后者，后者更重要且必要，因为先有后者再有前者”。

### DC-006: 非平凡设计任务需要外部对标

对象：架构、产品化、公开项目、可复用机制、交互方案与文档体系等非平凡设计任务  
约束：设计开始或收束前，应查找近似方案，整理它们相对当前设计的优点、可吸收点和不吸收原因；若不进行外部对标，应说明任务太小、用户要求离线，或已有足够本地依据。  
原因：设计任务容易闭门造车；外部对标可以暴露成熟方案的优点，降低重复试错。  
检查方式：相关输出应能说明“近似方案是什么、它们强在哪里、当前方案吸收什么、不吸收什么及原因”。  
状态：`active`  
来源：用户要求“做设计时会联网查找是否有近似内容，并整理相对于自身设计的他们的设计优点”。

### DC-007: CCA 系统入口必须索引化

对象：`references/cca/system.md` 与 `references/cca/system/`  
约束：`system.md` 只承接轻量索引、默认启动路径、topic map 和硬边界；细则必须拆入专题文件，并且只在命中具体触发时读取。  
原因：全量系统正文作为默认必读文件会增加 token 成本和响应延迟，也会削弱规则遵循。  
检查方式：默认启动应只需要 `system.md`、正式状态四件套和当前活跃层文档；若读取专题文件，必须能说明触发原因和读取目标。  
状态：`active`  
来源：用户指出 `references/cca/system.md` 过重是大问题，要求参考 Claude 的轻量入口与按需展开思路。

### DC-008: 负载成本是一等设计约束

对象：ACH 默认启动、`cca` 读取顺序、切窗恢复、GitHub 示例
约束：token、读取量、启动延迟、手工交接和重复说明都应计入设计成本；不能用全量读取、长交接或要求用户反复补背景来维持系统成立。
原因：ACH 的核心价值是降低长任务失去连续性的成本；如果系统本身把读取和恢复成本转嫁给用户，就会抵消该价值。
检查方式：默认路径是否只读取最小必要状态；是否能说明每次展开的触发；切窗是否先补状态再给短启动语；是否避免让用户手工携带长摘要。
状态：`active`
来源：天网、adg+cca 与 ACH `system.md` 轻量化讨论中反复出现的上下文成本问题。

### DC-009: 首页表达先讲失败模式和试用路径

对象：README、docs、examples、发布文案
约束：公开表达优先讲 ACH 防止什么失败、用户如何开始试用，再解释 guard-mode / continuity-mode 等内部结构。
原因：用户先判断项目是否解决自己的问题；过早讲内部结构会提高理解成本，削弱 GitHub 首页转化。
检查方式：首页首屏应出现明确失败模式、最小调用例子和正式名/调用缩写关系；内部模式说明应放在问题和试用入口之后。
状态：`active`
来源：ACH 高星化与首页收紧讨论。

## 更新规则

新增设计约束前，先判断它是：
- 层级/分支/产物约束
- 普通表达意见
- 当前轮执行要求
- 历史参考

只有第一类进入本文件。
若新约束与旧约束冲突，不直接覆盖；先按 `outer.md` 的效力状态标记为 `pending`、`superseded` 或 `rejected`。
