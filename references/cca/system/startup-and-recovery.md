# 启动与恢复

仅当当前任务需要启动绑定、恢复、交接、跨窗口延续，或检查最小启动包是否足够时，读取本文件。

## 最小启动包

`cca` 默认启动包是：

- `references/cca/system.md`
- 已绑定任务的状态文件：
  - `current-goal.md`
  - `confirmed-constraints.md`
  - `pending-items.md`
  - `decisions.md`
- 当前活跃层文档：
  - `references/cca/inner.md`
  - `references/cca/outer.md`

`references/cca/system/` 下的专题文件不属于默认启动包，只按触发读取。

## 启动顺序

1. 读取 `references/cca/system.md`。
2. 找到工作区根 `.cca-bindings.json`。
3. 判断当前任务是否已有正式状态根。
4. 若有状态根，读取四件套并判断活跃层。
5. 若无状态根但 continuity 确有必要，先创建或绑定最小正式状态根。
6. 读取活跃层文档。
7. 只有活跃决策需要时，才展开一个专题文件。

## 首次绑定

当不存在有效正式状态根，且当前任务确实需要 continuity 时：

- 选择稳定的 `task_key`
- 创建 `.cca-state/<task>/`
- 创建四个状态文件
- 创建 `state-manifest.json`
- 将绑定写入工作区根 `.cca-bindings.json`
- 记录为什么当前任务需要正式 continuity

不得为简单、局部、低风险任务创建正式状态根。

## 恢复失败

如果状态根缺失、不完整、过期或互相矛盾：

1. 点名缺失或冲突的状态
2. 优先修复或更新正式状态根
3. 只读取修复所需的来源证据
4. 不用广泛回读系统母本替代状态修复

恢复失败首先是状态质量问题，不是默认加载全部 reference 的理由。

## 跨窗口延续

切窗或准备新对话前，产出最小交接包：

- 当前目标和阶段
- 后续仍生效的已确认约束
- 影响下一步判断的未决项
- 下个窗口必须保留的决策
- 活跃层和下一步动作
- 正式状态根路径，或明确无状态根的原因

交接包指向正式状态，不替代正式状态。

切窗交接的目标是恢复现有状态和阶段位置，不是只给一句“继续下一步”。若用户隔一段时间后回来，交接包也应足够帮助其理解当前停在哪里、为什么停在这里、下一步从哪里继续。

面向用户的新窗口启动语应尽量短，但短句背后的状态必须已经落地；不得通过要求用户手工携带长摘要来弥补状态缺失。

## 交接摘要边界

交接摘要可以压缩到达当前状态的路径，但不得：

- 把未解决假设写成确认事实
- 覆盖四件套
- 成为未来决策的唯一来源
- 默认携带完整 transcript 历史

后续窗口若需要来源证据，应读取状态文件或交接包点名的窄证据。

## 上下文限制规则

当对话历史、工具输出、网页内容、代码片段或审查 transcript 已经挤占当前决策时，应把持久状态移入正式状态根。

外化完成后，完整历史降级为冷证据。后续默认读取状态文件和被点名的局部证据，而不是反复全盘读历史。

## 完成检查

启动或恢复只有在后续 agent 能从以下内容继续时才算完成：

- 系统索引
- 正式状态根
- 活跃层文档
- 被明确点名的证据

若继续推进仍依赖未写出的聊天记忆，则启动或恢复未完成。
