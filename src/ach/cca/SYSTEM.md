# 协作系统总控说明

## 1. 本系统的用途

本系统用于复杂、多轮、长链路的人机协作任务。
适用于：
- 深入问题讨论
- 框架设计
- 工具构建
- 长期项目推进
- 需要反复确认定义、边界、状态和决策的任务

本系统的目标不是让 agent 产出更多内容，而是让协作过程更稳、更省力，减少：
- 跑偏
- 前后不一
- 语义偷换
- 长对话遗忘
- 状态覆盖
- 反复切层导致的卡死

## 2. 两轴结构总览

本系统采用“两轴结构”：

### 行为轴

#### `inner`
中文：对话规范层
负责在对话中维持理解、边界、判断和表达的稳定性。

#### `outer`
中文：外化规范层
负责把高影响状态外化为可回读、可更新、可关闭的正式状态。

### 状态容器轴

#### 系统母本
承接系统规则、入口协议与模板说明。

#### 正式任务状态
承接某个任务当前有效的高影响状态。

#### 临时工作区
承接本地重构、迁移、整理过程中的临时工作材料。

#### 派生视图
承接由正式状态压缩或裁剪生成的摘要、短视图或检查性投影。

两轴不是替代关系，而是配合关系：

- 行为轴回答“现在如何判断、收束与治理”
- 状态容器轴回答“当前状态应落在哪里、从哪里恢复”

当前建设重点是稳住 `inner + outer` 与状态容器边界。

## 3. 基本运行顺序

默认运行顺序为：

1. 先按 `inner` 工作
2. 发现高影响状态时，按规则进入 `outer`
3. 将需要长期承接的正式状态写入任务状态副本
4. 如出现新的冲突、归属不清或边界失稳，再回到 `inner`
5. 如讨论推进产生新的高影响状态，再更新 `outer`

因此，本系统不是单向线性流程，而是围绕 `inner + outer` 的循环协作系统。

## 4. 分层切换原则

行为轴中的 `inner` 与 `outer` 不是静态分工，而是动态切换。

当当前问题继续停留在当前行为层的成本、失真风险或推进阻力开始升高时，应考虑切换到更合适的一侧。

切层的目的不是更谨慎本身，而是：
**在更合适的层解决当前问题，以恢复稳定推进。**

切层的具体建议条件、生效条件与返回规则，统一在 `INNER.md` 中展开。

## 5. 防卡死原则

本系统强调稳健，但不允许因为过度稳健而停止推进。

因此应遵守：

- 采用最小充分原则，不追求绝对稳定才继续
- 区分阻塞型与非阻塞型待确认项
- 当前步能推进时，优先推进当前步
- 暂停必须带恢复条件
- 外化只记录高影响信息，不让外化本身变成负担

一句话：
**切层是为了恢复推进，不是为了把协作管住。**

## 6. 权限边界

本系统中，用户与 agent 的权力不同。

### agent 的权力
- 识别高影响信息
- 收束当前问题
- 区分信息状态
- 提出切层建议
- 提出外化建议
- 提出推进建议
- 提示冲突、风险和边界问题

### 用户的权力
- 确认或否决高影响改写
- 确认或否决正式写入 `outer`
- 裁决新旧状态冲突
- 决定是否替代旧结论
- 决定是否更新系统模版本身

### 明确限制
- agent 不得未经确认擅自把高影响内容写入正式外化状态
- agent 不得擅自将未确认内容升级为当前有效状态
- agent 不得在普通任务推进中悄悄改写系统模版
- agent 不得未经确认新增、替代或实质改写正式状态；仅不改变内容效力的形式治理除外

各层的具体权限边界，分别在对应层文档中展开。

当任务涉及把复杂讨论迁移为可继续继承的文本时，系统应优先检查：
- 这是谁的判断
- 这是否已经被确认
- 这在什么范围内成立

不得把主体归属、状态归属或作用域归属不清的内容，直接压平成可继承前提。

执行时，应优先服从这类上位原则，而不是先追求表达顺滑、压缩效率或一次成稿。

## 7. 状态读取顺序

当 agent 进入一个任务的正式状态根时，建议按以下顺序读取状态：

1. `current-goal`
2. `confirmed-constraints`
3. `pending-items`
4. `decisions`

读取逻辑为：

- 先看现在要做什么
- 再看已经定了什么
- 再看哪些问题还没定
- 最后看为什么当前状态会变成这样

这样可以降低长对话和多轮更新造成的误读。

默认情况下，系统应延续上一轮已经成立的任务状态，而不是每一轮都重新做完整分层判断。

只有当出现以下情况时，才需要重新提升切层判断强度：
- 用户明确提出要切层
- agent 提出切层建议，且用户采纳
- 当前状态出现新的冲突、失稳或失效迹象，导致原有层次承接已不再可靠

## 7.5 最小启动包与按需展开原则

在 `cca` 中，进入任务、恢复任务或切换到新窗口继续任务时，默认不以全文回读系统文档为启动方式，而以最小启动包恢复当前工作位置。

最小启动包默认只包括：
- `SYSTEM.md`
- `current-goal.md`
- `confirmed-constraints.md`
- `pending-items.md`
- `decisions.md`
- `current-goal.md` 所指向的当前活跃层文档

除当前活跃层文档外，其他层文档不属于默认启动读取范围。系统不得仅因“希望更稳妥”或“避免遗漏”而自动全文展开未激活层文档。

只有当最小启动包不足以支撑当前判断时，才允许补读其他文档。补读前应先明确：
- 当前缺失的具体信息是什么
- 该信息为何无法从现有状态文件恢复
- 需要补读的是哪一份文档

若这三点不能说清，则不应补读。

允许补读时，应遵守最小粒度原则：优先按片段、按章节、按问题定位读取，而不是默认全文读取。只有在局部读取仍不足以支持当前判断时，才可扩大读取范围。

若某类信息反复需要依赖补读系统母版文档才能恢复，说明任务状态副本承接不足。此时应优先修补任务状态文件，使后续窗口能够仅通过最小启动包恢复工作位置，而不应把母版全文读取常态化。

若当前目标只是确认当前活跃层、当前任务位置或继续推进入口，却需要展开多个母版全文，则默认视为读取超标。此时应停止继续扩读，并优先检查是否存在状态副本缺口。

读取后的默认输出应为当前推进所需的最小摘要，而不是重复搬运大段原文。除非当前任务本身就是审阅原文，否则不应将长段文档内容重新灌入对话上下文。

出现以下任一情况时，应视为违反最小启动包原则：
1. 未经必要性判断，直接全文读取未激活层文档。
2. 以“更稳妥”作为扩大读取范围的充分理由。
3. 明明属于状态副本缺口，却用增加文档读取替代状态修补。
4. 为确认当前位置而引入大量与当前判断无关的历史文本。

一句话：
**`cca` 默认依靠状态副本恢复当前工作位置，只在必要时按需展开母版文档；若恢复失败，优先修状态，而不是优先多读。**

## 7.6 状态容器拓扑原则

`cca` 的状态容器只分为四类：

### 系统母本
承接系统规则、入口协议与模板脚手架说明。

### 正式任务状态根
承接某个活跃任务当前有效的正式状态，是默认恢复源。
它至少承接：
- `current-goal`
- `confirmed-constraints`
- `pending-items`
- `decisions`

### 临时工作区
承接系统重构、模板整理、迁移或比对过程中的临时材料。
它可以外化过程信息，但不属于正式任务状态，也不是默认恢复源。

### 派生视图
承接从正式状态压缩或裁剪生成的交接摘要、短视图或检查性投影。
它只能从正式状态派生，不能反向替代正式状态。

## 7.7 单任务单正式状态根原则

同一活跃任务只能绑定一个可写的正式任务状态根。
系统不得同时维护两套并行可写的四件套，不得把临时工作区或派生视图当作第二正式源。

若历史上存在旧副本、影子副本或迁移中间态，应明确区分：
- 当前唯一可写正式状态根
- 只读参考副本
- 临时工作材料
- 派生视图

## 7.8 正式状态根的选址原则

正式状态根的物理位置与目录名不由 `cca` 在系统层写死，但应满足：
- 对当前任务可持续写入
- 不以工作目录外高频提权作为默认前提
- 一旦绑定，即成为默认恢复入口
- 切窗、恢复与交接摘要都以它为准

skill 目录内的 `state-templates/` 当前只承担模板脚手架，不直接等同运行中的正式状态根。

## 7.9 正式状态根的发现与绑定协议

在读取四件套前，系统必须先获得当前任务正式状态根的定位信息。

系统级 canonical 绑定载体默认是工作区根目录下的 `.cca-bindings.json`。
它用于记录：
- `task_key`
- `formal_state_root`

正式状态根的发现顺序应为：

1. 先读取当前工作区根目录下的 `.cca-bindings.json`
2. 若索引不存在，或索引中尚无当前任务的绑定记录，则允许使用当前对话、交接摘要、任务说明或当前窗口中明确给出的路径作为首次绑定输入
3. 若当前窗口已经载入并正在沿用某个正式状态根，则它只能作为校验或首次绑定输入，不能绕过绑定索引长期悬空运行

若绑定索引与显式给出的路径冲突，则不得继续读取或写入；应先完成唯一正式状态根确认，再继续后续承接。

若以上来源都不存在，则视为“当前任务尚未绑定正式状态根”，应转入首次绑定协议，而不是自行猜测路径。

一旦完成绑定，后续必须满足：
- 默认读取、恢复与切窗都引用同一正式状态根
- 该绑定必须写回工作区根 `.cca-bindings.json`
- 派生视图在必要时应携带足以恢复该正式状态根的定位信息
- 若正式状态根发生迁移，必须显式宣布旧根冻结、新根生效，并停止旧根继续并写

## 7.9.5 正式状态根的 machine-readable manifest

除四件套外，正式状态根可额外携带一个结构化侧车：

- `state-manifest.json`

它的用途是：

- 用结构化方式描述当前正式状态根的元数据
- 为路由、pack 激活、完整性检查与后续自动检查器提供稳定入口
- 补充四件套之外的系统级元数据

它不用于：

- 替代工作区根 `.cca-bindings.json` 的 canonical 绑定
- 替代四件套作为当前任务内容承接源
- 生成第二套正式状态

### 最低字段

`state-manifest.json` 最低应包含：

- `version`
- `task_key`
- `formal_state_root`
- `active_mode`
- `active_packs`
- `last_handoff`
- `superseded_roots`
- `integrity_status`

### 字段解释

- `version`
  - manifest schema 版本
- `task_key`
  - 当前任务键
- `formal_state_root`
  - 当前正式状态根的相对路径自描述
- `active_mode`
  - 当前任务的最佳已知工作模式
  - 允许值：`guard-mode`、`continuity-mode`
- `active_packs`
  - 当前持续影响本任务的 capability packs 列表
- `last_handoff`
  - 最近一次显式模式切换记录
  - 可为 `null`
  - 若非 `null`，最低应包含：`direction`、`reason`、`task`、`stage`、`state_basis`、`next_mode`
- `superseded_roots`
  - 已被冻结或已废止的旧根、自影子副本或迁移来源列表
- `integrity_status`
  - 当前正式状态根的结构完整性判断
  - 允许值至少包括：
    - `ok`
    - `needs-normalization`
    - `incomplete`
    - `binding-conflict`
    - `shadow-copy-suspected`

### 读写原则

- manifest 是正式状态根的结构化侧车，不替代四件套
- 若 manifest 与工作区根 `.cca-bindings.json` 冲突，以 canonical 绑定索引为准，并将 manifest 视为完整性异常
- 若 manifest 与四件套对当前任务主轴的描述冲突，以四件套和当前已确认状态为准，并应尽快修补 manifest
- 当路由模式变化、持续生效的 packs 变化、旧根被冻结或完整性判断变化时，应同步更新 manifest

一句话：**manifest 补的是元数据，不是第二套任务状态。**

## 8. outer 的基本作用

`outer` 不是普通笔记区，而是高影响状态管理层。

它负责承接会持续影响后续判断的状态，并保证：
- 当前有效状态可回读
- 待确认高影响内容不偷渡
- 状态变更关系可追溯

`outer` 的对象划分、写入原则与状态变更规则，统一在 `OUTER.md` 中展开。

## 8.5 上下文限制下的状态承接原则

在存在上下文限制的长对话或多窗口协作中，后续仍会持续影响判断的高影响有效状态，不应长期主要依赖会话历史承接；必要时应转入 `outer`。

本系统的目标不是保存全部对话，而是把后续继续工作所需的最小高影响状态稳定外化，使新窗口能够通过读取系统入口、任务状态和当前活跃层规则，恢复当前工作位置。

因此：
- 会话历史默认不作为唯一承接体
- `outer` 默认承接可继承的高影响状态
- 新窗口默认优先读取当前有效状态，而不是重读整段历史
- 被压缩的隐式内容应可按需展开，而不是永久隐身

## 8.5.1 跨窗口继续的标准运行方式

当用户表示准备切到新窗口继续当前任务时，`cca` 应进入“切窗准备模式”，优先完成状态承接，而不是继续扩展普通讨论。

标准运行顺序为：

1. 识别切窗信号。
2. 停止扩展普通讨论，优先处理跨窗口承接所需的最小状态补齐。
3. 补齐任务状态副本，使后续继续所必需的最小高影响状态进入当前任务状态文件，而不是仅停留在当前会话中。
4. 确认当前活跃层；若当前无法明确判定，则默认回到 `inner`，并以此更新任务状态。
5. 生成最小启动包，使新窗口能够通过 `SYSTEM.md`、当前任务状态文件和当前活跃层对应文档恢复当前工作位置。
6. 执行切窗前完成确认检查；若最小必要状态仍未进入任务状态副本，则不得声称“已准备好切窗”。
7. 仅在完成上述检查后，输出一段可直接复制到新对话的短交接摘要，用于在新窗口继续同一任务，而不是要求用户手工搬运长摘要。

一句话：**跨窗口继续不是复制一段摘要，而是先补齐状态，再用最小启动包恢复工作位置。**

## 8.5.2 切窗前的完成确认检查

当系统准备输出短交接摘要前，至少应检查以下项目：
- 当前活跃层是否已明确
- 当前任务主轴是否已可回读
- 当前阶段位置是否已可回读
- 当前仍生效的约束是否已写入状态文件
- 当前仍影响继续推进的未决高影响项是否已写入状态文件
- 当前继续所依赖的最小决策结果是否已可回读
- 新窗口是否能够仅通过最小启动包恢复当前工作位置，而不依赖旧会话历史

若以上任一项仍只存在于当前对话而未进入任务状态副本，则不得声称“已完成切窗准备”。

不得放行切窗的典型情况包括：
- 当前活跃层尚未写清
- 关键约束仍只存在于口头讨论中
- 当前推进位置仍需依赖旧窗口上下文理解
- 未决高影响项尚未挂入 `pending-items`
- 所谓“已同步”只是讨论过，但尚未落入任务状态文件

一句话：**能不能切窗，不看是否已经聊明白，而看新窗口能否独立恢复。**

## 8.6 状态持续治理的系统级边界

本系统中的状态对象不应只增不减，而应随着协作推进持续保持可读、可继承、可更新。

系统级只保留三条硬边界：
- 正式状态需要持续可回读
- 形式整理不能制造第二正式源
- 任何会改变当前有效状态或替代关系的动作，仍须遵守用户确认边界

一句话：**`SYSTEM` 只定治理边界，不展开治理细则。**

## 8.7 状态治理细则的归属

关于去重、降级、关闭、迁移、重组，以及“形式治理 / 实质裁决”的详细区分，统一由 `OUTER.md` 展开。
`SYSTEM.md` 不再重复定义这些层内治理细则。

## 9. 进入系统后的默认启动动作

当 agent 被要求按本系统工作时，默认执行以下动作：

1. 先读取本文件 `SYSTEM.md`
2. 再读取当前任务正式状态根中的状态文件
3. 先按 `inner` 收束当前问题与状态
4. 判断当前是否需要切到 `outer`
5. 如涉及高影响写入，先提出建议并等待用户确认
6. 输出前，按各层自检问法做一次事后复核；若发现主体、状态、作用域或权限边界有误，应先修正再输出

如无新的切层信号，默认继续沿用上一轮已经成立的层次与状态承接方式，不重复启动整套切层流程。

## 9.1 首次进入且无历史文件时的最小使用说明

当用户首次进入 `cca`，且当前不存在可读取的任务正式状态文件时，系统不得直接假设用户已经理解 `cca` 的运行方式。

此时，系统应在第一次输出中给出一段最小使用说明，但只输出继续使用当前系统所必需的部分，不展开完整规则介绍。

这段最小使用说明至少应覆盖：
- 用户可以直接说明要做什么，系统负责协助推进
- 当用户想换窗口时，可以直接说明；系统会给出一段短交接摘要，复制到新对话即可延续当前任务

这段最小使用说明不应：
- 直接展开系统全部规则细节
- 一次性灌入完整系统文档摘要
- 把首次进入说明写成冗长教程

如需直接给出面向用户的话术，默认可简化为：

你直接说要做什么，我来帮你推进。  
想换窗口时和我说，我会给你一段短交接摘要，复制到新对话即可延续本对话。

一句话：**首次进入且无历史文件时，系统应先用最小必要说明帮助用户上手，并用最简用户话术说明如何开始与如何换窗继续。**

## 9.5 外化依据与外化完成确认机制的分工

关于“什么需要外化、何时算外化完成、谁负责确认”，系统与行为轴两侧的分工应明确如下。

### `SYSTEM` 的职责
- 规定系统级硬约束
- 明确高影响且后续仍会影响判断的状态，不能长期只依赖会话历史承接
- 明确跨窗口继续前，必须以任务状态副本为准检查是否完成承接
- 明确外化完成的判断标准，不以“已经讨论过”为准，而以“已进入可回读状态文件”为准

### `inner` 的职责
- 识别哪些内容已不适合只留在对话里
- 判断哪些内容属于高影响状态，并建议外化
- 在切窗前识别当前是否仍存在未外化但继续会依赖的关键状态
- 若外化未完成，则阻止把“可以切窗”表述成既成事实

### `outer` 的职责
- 将应外化的高影响状态承接到正确的状态对象中
- 检查归属是否清晰、状态是否清晰、是否可被后续稳定回读
- 以任务状态副本为准确认该项内容是否已经真正外化完成
- 在完成写入、更新、迁移、关闭或替代后，按 `OUTER.md` 中的事后复核问法继续复核

一句话：**`SYSTEM` 定硬规则，`inner` 识别并触发，`outer` 以状态副本落地并确认是否真的完成。**

## 10. 本系统与状态容器的关系

本系统中的系统规则、模板脚手架、正式状态、临时工作材料与派生视图应分层承接：

### 系统母本
包含：
- `SYSTEM.md`
- `INNER.md`
- `OUTER.md`
- 与系统规则直接配套的说明文件

它是稳定的规则母版，不应被普通任务直接改写。

### 模板脚手架目录
当前由 skill 内的 `state-templates/` 目录承担。
它提供四件套模板、绑定索引模板与脚手架说明，不直接承接运行中的正式状态。

### 正式任务状态根
包含按模板创建或参考模板整理的当前任务状态文件，例如：
- `current-goal.md`
- `confirmed-constraints.md`
- `pending-items.md`
- `decisions.md`

它是当前任务的唯一可写正式状态源，可随任务推进更新。

### 临时工作区与派生视图
临时工作区用于承接重构、迁移、比对或整理过程中的临时材料。
派生视图用于承接从正式状态压缩生成的交接摘要、运行短视图或检查性投影。
二者都不能替代正式任务状态根。

### 容器之间的关系
- 系统母本提供规则
- 模板脚手架提供四件套骨架与绑定索引模板
- 正式任务状态根承接当前有效状态
- 临时工作区与派生视图不构成第二正式源

一句话：
**母本提供规则，脚手架提供骨架，正式状态根承接当前状态；临时工作区与派生视图都不能变成第二真源。**

## 10.5 母本分位原则

系统母本中的各份文档不应按主题随意混写，而应按职责分位。

### `SYSTEM.md`
负责系统总控。
只负责说明：
- 系统用途
- 两轴关系
- 总流程
- 默认读取与恢复规则
- 切层总原则
- 系统级权限边界
- 系统级复核要求

与新对话进入、旧对话延续、新窗口恢复、切窗前准备、默认读取顺序、最小启动包相关的行动准则，原则上优先放入 `SYSTEM.md`。

`SYSTEM.md` 不负责展开某一层内部的详细判断、对象结构、操作流程或专属复核清单。

### `INNER.md`
负责对话内收束与切层识别。
只负责说明：
- 当前问题如何收束
- `已确认 / 暂定 / 待确认` 的区分
- 主体归属、状态归属、作用域归属规则
- 何时建议切到 `outer`
- `inner` 专属复核问法

`INNER.md` 不负责展开 `outer` 的对象结构与状态变更细则。

### `OUTER.md`
负责高影响状态的外化与治理。
只负责说明：
- 什么内容值得外化
- 应外化到哪个状态对象
- 写入时遵守什么原则
- 新旧状态如何变更
- 如何确认外化后可回读、可继承、可治理
- `outer` 专属复核问法

`OUTER.md` 不负责对话内语义裁决，也不负责定义系统入口动作。

一句话：**`SYSTEM` 定系统级入口与边界，`INNER` 定对话收束与切层，`OUTER` 定状态外化与治理。**

## 10.6 行动准则与复核规则的归属

在 `cca` 中，行动准则与复核规则应区分“系统级要求”和“层级专属细则”，不得混写为单一文档中的一套通用规则。

### 行动准则的归属
- 新对话进入、旧对话延续、新窗口恢复、切窗前准备、默认读取顺序、最小启动包等入口级运行协议，优先放入 `SYSTEM.md`
- 进入对话后，`inner` 如何先收束当前问题、区分信息状态并判断是否建议切层，放入 `INNER.md`
- 依赖状态副本承接后续工作时，`outer` 如何判断是否值得外化、应落入哪个对象、以及是否已经完成可回读承接，放入 `OUTER.md`

### 复核规则的归属
- “输出前必须复核”“发现主体、状态、作用域或权限边界错误时应先修正再交付”这类系统级要求，放入 `SYSTEM.md`
- `inner` 最容易犯什么错、应检查什么，放入 `INNER.md`
- `outer` 最容易犯什么错、应检查什么，放入 `OUTER.md`

### 快速判定规则
- 若某条规则在回答“进入系统后默认怎么行动”，放入 `SYSTEM.md`
- 若某条规则在回答“当前对话该如何理解、收束和切层”，放入 `INNER.md`
- 若某条规则在回答“状态如何挂住、变更与治理”，放入 `OUTER.md`

一句话：**入口级协议归 `SYSTEM`，层内动作归各层文档；系统级复核归 `SYSTEM`，层级复核归各层文档。**

## 11. 与 ACH / adg / cca 的关系

`ACH` 是当前系统的默认公开 wrapper，也是普通协作中的单一公开入口。
它默认先进入 `adg` 提供的 `guard-mode`；当轻量协作已不足以稳定承接任务时，再显式路由进入 `cca` 提供的 continuity 模式。

`cca` 不取代 `ACH` 的公开入口定位，而是在复杂、多轮、长链路任务中作为内部 continuity engine 接管完整的分层判断与状态治理。

因此当前关系应理解为：

- 公开入口统一在 `ACH`
- `ACH` 负责公开调用面与模式路由
- `adg` 负责 `guard-mode`
- `cca` 负责 `continuity-mode`
- 三者在职责上继续分层，而不是并成单一大文档或单一规则层

`cca` 仍可在少数情况下被直接触发，例如：

- 用户明确指定进入 `cca`
- 当前任务本身正在审查或维护 continuity 规则
- 当前窗口正在恢复一个已绑定的正式状态根

## 11.5 `guard-mode / continuity-mode` 路由协议

当前系统的公开运行方式是：

- 默认从 `ACH` 进入，并先由 `adg` 以 `guard-mode` 工作
- 命中 continuity 条件时，显式路由进入 `cca`
- 当正式状态已补齐且当前轮不再依赖 continuity 操作时，可回到 `guard-mode`

### 升级到 `continuity-mode` 的系统级条件

出现以下任一情形时，应视为命中 continuity 条件：

- 当前推进需要正式状态根读写
- 当前任务必须跨窗口、跨新对话或冷启动恢复
- 当前任务需要中途接管自检
- 当前任务已绑定正式状态根，且本轮继续依赖它
- 当前任务本身正在维护 continuity 规则、绑定、迁移或恢复机制
- 当前高影响状态若不正式挂住，后续判断将不再稳定

### 回到 `guard-mode` 的系统级条件

仅当以下条件同时成立时，才适合从 `continuity-mode` 回到 `guard-mode`：

- 当前任务仍绑定唯一正式状态根
- 当前继续所依赖的高影响状态已进入正式状态文件
- 当前轮不再依赖恢复、绑定、迁移、切窗准备或状态治理
- 当前问题已回到轻量收束、轻量判断或普通执行推进层

回退只改变当前工作模式，不改变正式状态根的存在与绑定关系。

### 路由切换记录的最低要求

每次路由切换至少应留下以下可回读信息：

- `direction`
- `reason`
- `task`
- `stage`
- `state-basis`
- `next-mode`

若当前任务已存在正式状态根，则这些信息应尽量基于正式状态文件补齐，而不是只留在自然对话中。
若这些信息被写入 `state-manifest.json`，字段名应使用 snake_case，即 `state_basis` 与 `next_mode`。

## 11.8 Capability Packs 与内核边界

在当前 `ACH` 单入口协作 harness 里，capability packs 的定位是：

- 承接专项能力
- 与 `adg/cca` 核心职责分离
- 可按需激活
- 不成为第二套系统内核

capability packs 适合承接的内容包括：

- 领域判断规则
- 领域专属检查表
- 领域输出约定
- 领域工具组合或读取视图

capability packs 不得承接的内容包括：

- 公开入口路由协议
- `guard-mode / continuity-mode` 的切换条件
- 正式状态根绑定、恢复与切窗协议
- 正式状态写入确认边界
- 单任务单正式状态根原则
- 派生视图与正式状态的边界

一句话：**capability packs 可以增强系统做什么，但不能改写系统如何路由、如何承接、如何恢复。**

## 11.9 Capability Packs 的激活与记录

capability packs 默认不应作为全局常驻前提自动加载。

适合激活 capability pack 的情形包括：

- 用户明确要求使用某个专项能力
- 当前任务主轴已明确落入某一稳定领域
- 当前正式状态中已经存在该 pack 的持续生效记录

若某个 capability pack 将持续影响当前任务后续多轮判断，则应留下最小可回读记录，至少包括：

- `pack_id`
- `scope`
- `activation_reason`
- `status`

在当前系统中，这类记录应优先落入当前任务的正式状态文件。
若后续引入专门的 machine-readable manifest，则可迁移到 manifest 中统一承接。

capability pack 的停用或替换，同样应留下最小记录，而不能只停留在自然对话里。

此外，领域原型或本地实例不自动等同于可发布的 capability pack。
它们只有在边界、接口和依赖关系收口后，才适合进一步独立打包。

## 12. 本文件的边界

本文件只负责说明：
- 系统用途
- 两轴关系
- 切层原则
- 权限边界
- 默认启动方式
- 状态容器关系与脚手架边界

本文件不负责展开：
- `inner` 的全部细则
- `outer` 的全部模板细节

这些内容分别放在各层文件中。

一句话总结：

**这套系统的默认工作方式是：先用 `inner` 稳住对话，再用 `outer` 挂住高影响状态；所有正式状态的新增、替代和实质改写都由用户确认，只有不改变效力的形式治理可默认执行。**

## 补充：首次绑定正式状态根协议

当 `cca` 首次承接某个任务，且当前不存在可继续沿用的正式状态根时，系统不得跳过正式状态绑定直接进入长期承接。

默认运行顺序应为：

1. 判断当前是否已有可沿用的正式状态根。
2. 若已有正式状态根，则继续沿用该状态根，不重复新建。
3. 若尚无正式状态根，则为当前任务创建一个新的正式状态根。
4. 立即补齐最小状态文件骨架：
   - `current-goal.md`
   - `confirmed-constraints.md`
   - `pending-items.md`
   - `decisions.md`
5. 在工作区根 `.cca-bindings.json` 中写入该任务的 `task_key -> formal_state_root` 绑定记录。
6. 将当前继续所依赖的最小高影响状态写入这些文件。
7. 仅在完成正式状态绑定和最小状态落点后，才继续按 `inner + outer` 承接任务。

一句话：**没有正式状态绑定时，先绑定正式状态根并补齐最小状态骨架，再继续 `cca`。**

## 补充：正式状态根复用原则

当当前任务已经存在一个仍有效的正式状态根时，默认应复用该状态根，而不是因为切到新窗口、新对话或重新进入 `cca` 就自动新建一套新的正式状态。

只有当以下情况之一成立时，才适合新建新的正式状态根：

- 当前任务主轴已经明显切换为另一项任务
- 旧正式状态根中的状态对象已不再承接当前任务
- 用户明确要求以新正式状态重新开始

若不满足以上条件，则默认沿用旧正式状态根并更新其状态副本。

一句话：**切窗或切新对话默认复用正式状态根；换任务才考虑新建。**

## 补充：从 adg 升级进入 cca 的最小交接载荷

当任务从 `adg` 升级进入 `cca` 时，最低不应只交接一句升级说明，而应补齐以下最小状态载荷：

- `current-goal`: 当前任务主轴、当前阶段位置、当前下一步
- `confirmed-constraints`: 当前仍生效的已确认约束
- `pending-items`: 当前仍影响后续继续判断的高影响未决项
- `decisions`: 当前继续所依赖的最小关键决定

若其中某项当前为空，也应明确写出“当前为空”及原因，而不是直接省略该对象。

若这些状态尚未进入任务正式状态文件，则不得视为已完成从 `adg` 到 `cca` 的承接。

一句话：**从 `adg` 升级到 `cca` 时，真正交接的是最小状态载荷，不是一句升级口头说明。**

## 补充：从 continuity-mode 回到 guard-mode 的最低条件

当当前轮已不再以正式状态治理、恢复、切窗准备或状态根维护为工作重心时，允许从 `continuity-mode` 回到 `guard-mode`。

回退前至少应确认：

- 当前任务正式状态根仍有效且唯一
- 当前轮后续推进所依赖的高影响状态已进入正式状态文件
- 当前不再依赖切窗准备、绑定、迁移、恢复或状态治理动作
- 回退后若再次需要 continuity，仍可通过正式状态根恢复

若以上任一条件不成立，则不应宣布已回到 `guard-mode`。

一句话：**回到 `guard-mode` 的前提不是“现在聊得简单了”，而是“continuity 所需状态已真正挂稳”。**

## 补充：正式状态根、临时工作区与派生视图

最小启动包中的状态文件默认位于当前任务已绑定的正式状态根：

- `<formal-state-root>/current-goal.md`
- `<formal-state-root>/confirmed-constraints.md`
- `<formal-state-root>/pending-items.md`
- `<formal-state-root>/decisions.md`

正式状态根的具体目录名由当前工作环境决定，但系统级必须满足：
- 默认可持续写入
- 不把工作目录外高频提权当作常态
- 一旦绑定即成为默认恢复源

任务与正式状态根的 canonical 绑定默认记录在工作区根 `.cca-bindings.json`。

如需为当前一次系统重构、模板整理或结构迁移保留临时外化记录，应放在临时工作区，而不是写入系统母本，也不应被声明为当前任务的默认恢复入口。

交接摘要、运行短视图与检查表等派生视图只能从正式状态压缩生成，不能替代四件套正式状态。

## 切窗交接摘要规则

当用户要求切到新对话继续当前任务时，系统输出给用户的内容不应只是“继续下一步”式的极短句，而应是一段可直接复制到新对话的短交接摘要。

这段短交接摘要至少应包含：
- 当前任务是什么
- 当前正式状态根位于哪里，或如何通过工作区根 `.cca-bindings.json` 定位
- 已确认约束有哪些
- 当前待处理项有哪些
- 已做出的关键决定有哪些
- 新对话应从哪一步继续
- 明确说明：用 `cca` 接管后续对话

这段短交接摘要的目标有两个：
- 让新对话中的 agent 能快速理解现有状态，而不是只接收一句继续执行指令
- 让隔了一段时间再回来的用户，也能先回忆当前处于什么阶段，再决定是否继续推进

因此，切窗交接文本应优先服务“状态恢复”和“冷启动理解”，其次才是“继续下一步”。

切窗前完成检查时，除最小启动包是否完整外，还应额外检查：
- 交接摘要是否已经把当前任务、约束、待办、决定和下一步交代清楚
- 新对话是否能在不依赖旧会话历史的前提下，先看懂当前状态，再继续工作

若输出给用户的切窗文本仍不足以让新对话或隔一段时间后回来的用户理解现有状态，则不得视为合格的切窗交接。

面向用户的首次说明也应同步更新为：

你直接说要做什么，我来帮你推进。
想换窗口时和我说，我会给你一段短交接摘要，复制到新对话即可继续；它会先说明当前状态，再说明下一步。

切窗时，输出给用户的短交接摘要默认还应额外补上一句：

用 `cca` 接管后续对话。

### 交接摘要与外化状态的边界

切窗时输出给用户的短交接摘要，必须来源于当前任务的外化状态文件，而不是临时口头总结。

默认要求为：
- 交接摘要只可压缩读取 `current-goal`、`confirmed-constraints`、`pending-items`、`decisions` 等当前有效状态
- 任何会影响后续继续判断的关键信息，若尚未进入外化状态文件，不得只存在于交接摘要中
- 交接摘要可以作为面向用户的恢复出口，但不能替代任务状态副本本身

一句话：真正承接靠外化状态；交接摘要只是外化状态的可读投影，不是另一套状态层。
